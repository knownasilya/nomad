# Unify the filesystem on Autobase behind a single `beaker.fs` API

**Status: Proposed.** Both feasibility gates passed on 2026-06-22: the blob-model spike (Q1, `app/scripts/spike-autobase-fs.mjs`) and the open-cost benchmark (Q2, `app/scripts/bench-drive-open-cost.mjs`). Autobase-only is therefore technically clear; the decision to actually adopt it now turns on *strategic* trade-offs (the rewrite cost vs. keeping Hyperdrive's mature checkout/ecosystem — option 2), not on feasibility. Phases 0–5 have substantially landed (2026-07-01 → 2026-07-02). The view-open + reducer + blob/content helpers live in one shared module (`shared/fs-core.mjs`, imported by both apps, guarded by golden-vector + version-parity tests); Q2's long-oplog caveat is **closed** (10k-op drive cold-opens flat, ~9 ms p50); Q6 is **resolved to a plain Hyperbee view**; the wire format is **v1** (`FS_FORMAT_VERSION = 1`: hybrid `{ metadata, blob, value }` records with file bytes in per-writer Hyperblobs cores, swept across both runtimes); **`beaker.fs` is now the single public drive API** — `beaker.hyperdrive`/`beaker.autobase` are removed and all ~63 userland call sites + the fg-process RPC layer run on `beaker.fs`; new drives are Autobase-from-birth; the serve path is unified; and the dead ADR-0006 migration + ADR-0007 aggregation code is deleted. **The one architectural reservation:** `beaker.fs` still dispatches internally to a retained Hyperdrive implementation for `hyper://private/`, profile, and external Hyperdrives — physically deleting that backend means re-homing the private/profile drives onto Autobase, which is gated on **Q4 (encryption parity)** and needs live-runtime verification. Remaining gates before *Accepted*: Q4 encryption, a real Autobase checkout rebuild (Q5), the private/profile re-homing, and **live-swarm** desktop↔mobile interop on v1 (byte-identical offline, not yet exercised over the wire). Until those land, this is a direction, not a commitment.

Today a `hyper://<key>/` URL resolves to one of two incompatible backends — a single-writer **Hyperdrive** or a multi-writer **Autobase/Hyperbee Collaborative Drive** — and the URL does not say which. Routing is decided by out-of-band local state (`type:'autobase'` in `/drives.json`, or whether a collaborative session is already loaded; see `bg/protocols/hyper.js`, `bg/filesystem/index.js getDriveIdentFull`). This proposal collapses that split: **every Drive is an Autobase from birth**, the Hyperbee view is extended to carry per-file **metadata** and a proper **blob** store, and userland programs against a single `beaker.fs.*` API. `beaker.hyperdrive` and `beaker.autobase` become thin deprecated shims over `beaker.fs`, then are removed.

This supersedes ADR-0007 (profile aggregation) and amends ADR-0006 (it removes the eager one-way Root-Drive→Autobase migration) and ADR-0008 (blogs-as-Autobase becomes the uniform default, not a special case).

## Why

The dual-backend decision is the single largest source of complexity in the codebase, and it metastasizes into surfaces that become unchangeable once third parties depend on them:

- **The public API is forked in two.** `beaker.autobase` is advertised as "mirroring" `beaker.hyperdrive` (`bg/web-apis/bg/ai.js`) but has no top-level `readFile`, a `diff()` that always returns `[]`, no `query()`, and silent-no-op `mkdir`/`updateMetadata`/`symlink` (`bg/web-apis/bg/autobase.js`). The first thing a developer builds — a backend-agnostic file helper — is exactly what breaks.
- **Wrong-backend opens hang instead of throwing**, so the code is littered with type-guards and timeouts; one missed guard reintroduces the hang (e.g. file-explorer writes still hardcode `beaker.hyperdrive.drive`).
- **Detection is unreliable for the drives that matter.** `isCollaborativeDrive` returns `false` for any remote drive not yet loaded — i.e. every drive a new user visits — so userland carries four separate "try Hyperdrive, fall back to Autobase, cache the guess" dances.
- **The Collaborative-Drive data model can't carry metadata.** The view is a bare Hyperbee storing file bodies inline (base64 for binary, `bg/hyper/autobases.js:247`); there is no `mtime`/`ctime` and no blob structure, so `stat` fabricates zeros and bodies bloat the append-only oplog.

The decisive argument for *Autobase-from-birth* specifically: **an Autobase's bootstrap key is stable for the drive's entire life.** A drive can go from one writer to many by `addWriter` without ever changing its URL. That single property dissolves two of our most awkward standing decisions:

- ADR-0006's **eager, one-way Root-Drive→Autobase migration that mints a new key** disappears — there is nothing to migrate because drives are already Autobases.
- ADR-0007's **writer-key aggregation** (per-device content drives + `writer-keys.json` + `Post.author.writerKey` merge) disappears — a Profile Drive just gains writers like any other drive, keeping its URL. (That aggregation path is currently write-only and unread anyway, so we delete unbuilt complexity, not working code.)

Done right, this also *restores* capability rather than only removing it: a metadata-carrying Hyperbee view makes `query()`, `diff()`, and `ctime` ordering work uniformly across all drives for the first time.

## Considered options

1. **Autobase-only behind `beaker.fs` (this decision).** Uniform model, one URL per drive for life, kills the migration and the aggregation hack, one API to document/type/teach. Cost: we must add blob + metadata support to the Autobase view (a multi-writer Hyperdrive, recomposed from the Hyperbee + Hyperblobs primitives Hyperdrive already uses — see Open question 1), accept Autobase open/linearization overhead on every drive including single-writer ones, and re-home versioned `checkout` (genuinely at risk) and the blob-serving path.

2. **Unify only the API; keep a hybrid backend.** Ship `beaker.fs` that internally dispatches to Hyperdrive for single-writer and Autobase for multi-writer, hiding the split from userland. Cheaper, keeps Hyperdrive's efficient blobs/checkout/range for the common single-writer case. **Rejected as the end state** because it does *not* deliver the killer property: a drive born single-writer is a Hyperdrive, so going multi-writer still mints a new key and breaks its URL — ADR-0006's migration and ADR-0007's aggregation both survive. It remains a viable *transitional* shape (see rollout) and a fallback if the blob spike fails.

3. **Status quo (two backends, two APIs).** Rejected: the leak is now codified in `CLAUDE.md` as a permanent law ("two read paths, not interchangeable"), which is the tell that it has been accepted rather than contained.

## Consequences

Positive:
- One drive backend, one wire format, one `beaker.fs` API. The dts and the AI reference describe one surface (and should be generated from the manifest, not hand-synced).
- A drive's URL is permanent; single→multi-writer is additive. ADR-0006 migration and ADR-0007 aggregation are removed.
- `query`/`diff`/`stat` with real `ctime`/`mtime` work on every drive; the bookmark format and explorer sort stop being second-class on collaborative drives.

Negative / accepted costs:
- **Autobase-backed drives are not Hyperdrives on the wire.** Standard `hyper://` clients (hypercore-fetch, Agregore, Beaker, Pear) cannot read them. We already took a clean break from the v10 ecosystem (ADR-0001), so this is consistent, but it makes Nomad's `hyper://` a Nomad-only dialect. State this explicitly in user-facing docs.
- **Every drive pays Autobase cost** — an oplog, an indexer, linearization, and a warm-up before the view is readable (today's 15s replication-poll loops hint at the latency). Single-writer drives that used to be a plain Hyperdrive now carry that weight.
- **Blob-bytes-through-the-oplog is avoidable but easy to get wrong.** If blobs are written inline into the linearized view during `apply()`, a file's bytes traverse the append-only oplog and are retained forever. The recommended design (per-writer Hyperblobs cores, Open question 1) avoids this entirely — only a small pointer goes through the oplog — but it is the more involved option, and the cheap-but-wrong inline path is the tempting default.
- **Lost-until-rebuilt Hyperdrive features:** versioned `checkout` (`hyper://key+version/`, wired into `bg/ui/permissions.js`, `bg/web-apis/bg/shell.js` import/export, `bg/ui/tabs/pane.js` watch) — this one is genuinely at risk (Open question 5). Direct HTTP **Range** streaming (`bg/protocols/hyper.js:406,624`) is *not* really lost: Hyperblobs supports ranged `get()` and `hypercore-blob-server` serves blobs over local HTTP with range requests, so the serve path is re-pointed, not rebuilt.
- **The consensus-critical `apply()` is duplicated across two runtimes** — desktop `app/bg/hyper/autobases.js` (Electron/Node) and mobile `mobile/backend/lib/drive-manager.mjs` (Bare). As of 2026-07-01 both live in **one repo** (`nomad/mobile/` was moved in), which removes the *temporal* drift risk — a wire-format change now touches both in a single commit/PR. But it is **not** npm workspaces: still three `package.json`/`node_modules` trees and two JS engines, so version-parity of the shared P2P deps stays manual and the Bare-vs-Electron seam remains. Extending the value/op shape still changes a function that must stay byte-identical or replication fails with `DECODING_ERROR: Unknown wire type`. Extract it into one shared module (now a plain relative import, not a published package) **before** the shape changes, guarded by a golden-vector test.

## Open questions (spikes that gate acceptance)

1. **Blob storage — the crux, resolved in shape by Hyperblobs + the Autobase model.** Hyperdrive is itself a **Hyperbee** (path→metadata index) plus a **Hyperblobs** content store (its own `package.json` depends on both); `hyperblobs` is *already* in both repos' trees transitively (v2.12.1). So "multi-writer Hyperdrive" means recomposing two primitives we already ship over Autobase, not inventing storage. Hyperblobs chunks large binaries across a Hypercore, `put()` returns an id `{ byteOffset, blockOffset, blockLength, byteLength }`, and `get(id[, range])` reads it back — exactly the `entry.value.blob` shape our serve path already consumes (`bg/protocols/hyper.js:621`). Autobase's `open(store)` can return *any* view structure (composite cores, a Hyperbee, or a HyperDB), so the architecture is free at the view layer; the binding constraint is on `apply`, which the Autobase docs require be a **pure, replay-safe reducer** — *"mutate only the view… do not open additional cores or perform side effects inside apply… side effects belong outside the linearization path"* — and which Autobase **re-runs on reorg**. That single rule decides the integration pattern:
   - *(b, the design) Per-writer Hyperblobs content cores.* Each writer owns a `blobs` Hypercore in its own namespace and `put()`s file bytes there **outside `apply`** (single-writer append — fast, native ranged reads, **no oplog bloat**). The Autobase op carries only a pointer `{ writerKey, blobId }`; `apply` does nothing but record `{ metadata, blob:{writerKey, blobId} }` in the view — an idempotent put, so it survives Autobase's re-apply/reorg unchanged. A reader resolves by opening that writer's blobs core (already replicated via the shared corestore/swarm) and `get()`ting the range. This cuts the stack along its natural seam: path-addressing and metadata live *centrally* in the shared view; each writer's content store needs no path semantics — exactly the case the Pear docs say to use Hyperblobs, not a Hyperdrive, for ("if you do not need path-addressed files shared between peers, prefer Hyperblobs over Hyperdrive"). *Spike target:* confirm a reader can resolve and range-read **another writer's** blobs core through the Autobase's corestore, including when that writer is offline but its data has replicated.
   - *(a, rejected on the model) View-owned blobs core written in `apply`.* Having `apply` write bytes into a `store.get({name:'blobs'})` view core seems tempting (Hyperblobs uses *named* cores, unlike Hyperdrive's keyed `name:null` blobs core that the current `_openFn` comment says "throws" under AutoStore). But it puts a **non-idempotent append** (Hyperblobs `put`) inside the reducer, and Autobase re-runs `apply` on reorg — replays would double-append and diverge the view (`DECODING_ERROR`). It also violates the docs' explicit "no side effects in apply." Rejected.
   - *(c, fallback) Small-files-only.* Skip Hyperblobs; cap drives at text/structured data. Cheapest; viable if drives are overwhelmingly sites/JSON, but forecloses media.
   - **Decision rule:** build (b) and benchmark a ~50 MB asset for write + ranged read + cross-writer resolve. If the cross-writer/offline resolve works, proceed with Autobase-only. If it proves unreliable, fall back to option 2 (unified API, hybrid backend) and keep Hyperdrive for blob-heavy single-writer drives; or, as a stopgap, (c). For local `<video>`/`<img>` range streaming, adopt `hypercore-blob-server` (the one new dependency) or adapt the existing `bg/protocols/hyper.js` Range path to `hyperblobs.get`.
   - **Result (2026-06-22): PASSED.** `app/scripts/spike-autobase-fs.mjs` builds two Autobase writers (autobase 7.28.1, corestore 7.10.1, hyperblobs 2.12.1) on the proposed view (`Hyperbee` of path→`{metadata, blob}`, pure keyed-put reducer) and confirms 10/10 checks, including a 50 MB blob: writer A puts bytes into its own Hyperblobs core *outside* `apply` and appends a `{core, id}` pointer; B resolves it cross-writer (full + ranged `[1MB,+64KB)`); B still serves it after replication is **cut** (offline-but-replicated); a never-downloaded blob is correctly **unavailable** while A is offline; and after a **partition → concurrent writes → heal**, both writers re-run `apply` on the reorg and **converge to byte-identical views** with all pointers still resolving both directions. Non-flaky across repeats. *Caveats:* in-process pipe replication (not Hyperswarm), one runtime/version (not the desktop↔mobile cross-repo case), plain Hyperbee view (not HyperDB, Q6), and no encryption (Q4). Those remain open.

2. **Open cost at browser scale — measured 2026-06-22, does NOT block autobase-only.** `app/scripts/bench-drive-open-cost.mjs` (local, no swarm; single-writer Hyperdrive vs single-writer Autobase; correctness-asserted). Autobase costs a consistent **~5–9× multiplier** but the absolute numbers are small: cold-open p50 4.6/4.9/6.3 ms and p95 7.7/7.7/10.0 ms at N=30/60/120 (Hyperdrive is a flat ~0.7 ms), and RSS ≈ **0.25 MB/drive** (≈29 MB for 120 drives). Even 120 drives opened *serially* is <800 ms, and a browser opens lazily/concurrently (only visible tabs), so real startup cost is far lower. Hyperdrive per-drive open is flat O(1); Autobase creeps mildly super-linear (4.6→6.3 ms as N goes 30→120), negligible below a few hundred drives. **Caveat not yet tested:** all drives had a 1-op oplog. Autobase open loads the persisted checkpointed view and applies only ops since the checkpoint, so it *should* stay flat, but a long-history drive (10k+ ops) must be benchmarked to confirm open doesn't degrade with edit count. Conclusion: performance is not a reason to reject autobase-only; the remaining reasons to prefer option 2 are strategic (rewrite cost, Hyperdrive's mature checkout/ecosystem), not speed.

3. **Remote read when the only writer/indexer is offline.** Confirm a single-writer Autobase's linearized, indexer-signed view is fully servable by any peer that has replicated it, with no live writer — the property Hyperdrive gives for free. (Memory/notes say remote Autobase opens read-only fine; verify for the *sole-writer-offline* case specifically, since it is the common P2P scenario.)

4. **Encryption parity.** Hyperdrive supports per-drive encryption keys; collaborative-drive pairing currently may omit an `encryptionKey`. Define one uniform encryption story for `beaker.fs` drives.

5. **History/versioning — the feature most likely to be cut.** Autobase exposes **no view checkout API** (`base.view` is only the current materialized state; the docs confirm there is no `base.view.checkout()`), and it rebases on reorg, so a stable `hyper://key+version/` point-in-time read is not free. Any history would have to be built on the underlying view core's own versioning *and* made robust to re-linearization. Decide explicitly whether to rebuild it (`bg/ui/permissions.js`, `bg/web-apis/bg/shell.js` import/export, `bg/ui/tabs/pane.js` watch all consume `getDriveCheckout` today) or drop it before removing the Hyperdrive path.

6. **View/index layer — raw Hyperbee vs HyperDB.** The metadata/index side of `beaker.fs` can be a bare Hyperbee (path→record, as today) or **HyperDB** (`holepunchto/hyperdb`, the newer schema-driven P2P database with typed collections and **secondary indexes**; `Hyperdb.bee(store.get({name:'db'}), def)` as the Autobase `open`). HyperDB is attractive because secondary indexes solve, natively, several problems this ADR exists to fix: `list()` has no `ctime`, `query()`/`diff()` can't span Autobase, and bookmarks-by-`href` / contacts-by-tag / post enumeration are manual scans today. It also replaces untyped-JSON-in-file-bodies with typed records. **Costs that keep it a candidate, not a default:** (i) Autobase's `open` may return any structure, so the *Autobase* side supports a HyperDB view; the unproven part is narrower — whether HyperDB itself tolerates Autobase **re-running `apply` on reorg** (its writes must be replay-safe and idempotent like a keyed Hyperbee `put`, and its secondary-index maintenance must survive re-linearization). Undocumented in its README; prove in the spike, with `pear-hyperdb` as a reference; (ii) it sharply raises the desktop↔mobile parity burden — both repos must match the HyperDB version **and the generated schema definitions byte-for-byte** or replication fails with `DECODING_ERROR`, which makes the Phase-0 shared-module extraction mandatory and adds a schema-codegen build step (akin to the existing `gen-schema-dts`); (iii) it is younger and faster-moving than Hyperbee, and the wire format would ride on it. **Decision rule:** prototype the spike's view both ways. Adopt HyperDB only if (a) it runs cleanly as an Autobase view and (b) the schema/codegen can be shared as one artifact across both repos; otherwise use a plain Hyperbee with a small hand-rolled index convention and revisit later.

   - **Result (2026-07-01): HyperDB is technically viable as an Autobase view — spike PASSED, but the recommendation is to STAY ON HYPERBEE for now.** `app/scripts/spike-hyperdb-view.mjs` (hyperdb 6.7.0; def generated by `app/scripts/build-hyperdb-def.cjs` — a `Hyperschema` + `hyperdb/builder` codegen step) builds the Autobase view as `HyperDB.bee(store.get({ name: 'db' }), def, { autoUpdate: true })`, with a `@fs/files` collection and a **non-unique secondary index** `@fs/files-by-tag`. `apply()` stays a pure reducer — typed `insert`/`delete` + `host.addWriter`, then one `view.flush()`. It confirms 7/7 checks, non-flaky across repeated runs, including the load-bearing one: after a **partition → concurrent tagged writes on both writers → heal**, Autobase **re-runs `apply` on the reorg** and both HyperDB views **converge byte-identically**, the **secondary index is still correct and consistent on both sides** (`tag=red → [/a1.txt, /from-a.txt, /from-b.txt]` on both), and a subsequent **delete removes both the record and its index entry** after replay. So condition (a) — "runs cleanly as an Autobase view, replay-safe incl. secondary indexes" — is **met**; HyperDB's `insert`/`delete`/index bookkeeping is deterministic under re-linearization, contradicting the README's silence being a red flag.
     - **Why Hyperbee is still the Phase-1 recommendation.** The blocker was never (a); it is **(b), the parity/coupling cost, which the spike did not lower and arguably confirmed.** HyperDB requires a **generated `def` (schema + index code) that must be byte-identical across the desktop and mobile trees** — the `hyperdb-spec/db/index.js` + `messages.js` this spike generates would become part of the consensus-critical wire surface, on top of the library-version pin. That is a *second* generated artifact to keep in lockstep across two `node_modules` and two runtimes (Bare/Electron), enforced only by the same manual pinning the golden-vector test guards for Hyperbee — and HyperDB is younger and faster-moving, so the wire format rides on a moving target. Against that, the Phase-0 shared reducer over a **plain Hyperbee** is ~40 lines with **no codegen**, and the concrete wins HyperDB would buy (ctime ordering, `bookmarks-by-href`, `contacts-by-tag`, post enumeration) are all reachable with a **small hand-rolled index convention** — e.g. secondary-index keys written as extra Hyperbee entries under an `/.idx/<field>/<value>/<path>` prefix by the same pure reducer. That keeps one wire format, one artifact to pin, and defers the schema-codegen coupling until there is a feature that genuinely needs typed collections + native indexes.
     - **Recommendation:** **Phase 1 uses a plain Hyperbee view** (extend the existing `{ metadata, blob }` record shape; add a hand-rolled `/.idx/…` convention if/when `query()`/`diff()` need it). **Keep HyperDB as a proven, revisitable option** — the spike shows the migration is a `open()`-swap + a shared codegen artifact, not a research risk — to be reconsidered once (b) is worth paying for (e.g. many secondary indexes, or typed-record validation becomes a first-class need). This resolves Q6 in favor of Hyperbee **on coupling cost, not on capability.**

## Migration & rollout

No production users exist, so we take a clean break (consistent with ADR-0001) — no data migration path is built.

- **Phase 0 — De-risk. DONE (2026-07-01).** Spikes 1–2 passed (blob resolution 2026-06-22; open-cost incl. the 10k-op long-oplog caveat now closed). Spike 6 ran the view layer both ways: HyperDB is replay-safe as an Autobase view but Q6 resolves to a **plain Hyperbee** on coupling cost. The shared `apply()` + view-open are extracted into **`shared/fs-core.mjs`** — a dependency-injected module (no bare P2P imports, since the repo-root `node_modules` has none; each app passes its own resolved `Hyperbee`/`b4a`) imported by both `app/bg/hyper/autobases.js` and `mobile/backend/lib/drive-manager.mjs` via relative path, behavior-preserving (current binary-Hyperbee wire format, `FS_FORMAT_VERSION = 0`). It is guarded by `tests/unit/fs-core-golden.test.js` (committed sha256 of the linearized view) and `tests/unit/fs-core-parity.test.js` (app vs mobile identical version pins). Still open at spike level: 3 (sole-writer-offline remote read). Gate: **met** — blob design chosen, view layer chosen (Hyperbee), open-cost acceptable at all edit counts. ⚠️ CI (`.github/workflows/release.yml`) runs `npm run build` but **not `npm test`** — add a test step so the golden/parity net actually gates PRs before the Phase 1 wire change.
- **Phase 1 — Wire format. DONE (2026-07-02).** `FS_FORMAT_VERSION = 1`. The view is a Hyperbee (`valueEncoding: 'json'`) of path → a **hybrid record** `{ metadata: { mtime, ctime, executable? }, blob: { core, blockOffset, blockLength, byteOffset, byteLength } | null, value: <base64> | null }`:
  - `blob` — a Hyperblobs id **plus `core`** (the hex key of the owning writer's blobs core; **not** `writerKey` as first sketched — a reader can't derive the blobs key from the writer key, so the resolvable core key is stored). File **content** is written to that per-writer Hyperblobs core **outside `apply`**; only the pointer enters the oplog (solves oplog bloat).
  - `value` — base64 bytes stored **inline** in the view, for small control records (`index.json`, `/.vault/*`, bookmarks, `drives.json`) and — necessarily — the writer-records `apply` itself authors on `addWriter` (a pure reducer can't write a blob). This **hybrid** shape is a deliberate amendment to the record first sketched here: it keeps manifest/control reads from having to replicate a peer's blob core, and it is the only shape compatible with the pure-reducer invariant. Both `null` = empty file.
  - Op shape: `{ op:'put', path, metadata, blob?, value? }` / `{ op:'del', path }` / `{ addWriter, profileUrl? }` / `{ removeWriter }`. `mtime`/`ctime` are **stamped by the producer** (apply has no clock); the put path preserves `ctime` from any prior entry (real ctime ordering — the capability this restores).
  - The one reducer + the blob/content helpers (`makeMetadata`, canonical `putBlob` pointer, `resolveBlob`, `readContent`) live in `shared/fs-core.mjs`; **byte-order determinism** of `metadata`/`blob` objects is enforced through those builders (they're stored verbatim in the signed view). Every producer/consumer across **both** runtimes was swept to v1 in one clean break (`bg/web-apis/bg/autobase.js`, `bg/hyper/autobases.js`, `bg/hyper/vault.js`, `bg/filesystem/index.js`, `bg/protocols/hyper.js` — now with **Range** on blobs; `mobile/backend/lib/drive-manager.mjs`, `mobile/backend/lib/vault.mjs`). Golden vector regenerated; the app and mobile dependency trees produce a byte-identical v1 view (verified offline). *Still gated for Accept:* live-swarm cross-writer blob resolution (Q1's cross-writer/offline case is spike-proven, not yet exercised in the running apps).
- **Phase 2 — `beaker.fs` API. DONE (2026-07-02).** New `manifests/external/fs.js` + `bg/fs.js` + `fg/fs.js`, exposed as `beaker.fs` (registered `secureOnly`). It implements `getInfo/entry/stat/get/readFile/list/readdir/query/diff/put/writeFile/del/unlink/mkdir/rmdir/copy/rename/watch` with **real** `'json'` read encoding (closing the documented-lie gap) and **real** `stat` (mtime/ctime/size from Phase 1). It is a **facade** that detects a URL's backend and delegates to `bg/hyperdrive.js` or `bg/autobase.js` (with a try-both fallback for reads of undetected remote drives), so userland gets one code path; Autobase-only (deleting the dispatch) is Phase 3. The dts (`beaker-dts.js`) and AI reference (`NOMAD_API_REFERENCE`) and the nomad.dev doc (`beaker.fs.md`) were **hand-written** for now — manifest-driven codegen (mirroring `gen-schema-dts.mjs`) is deferred to a follow-up.
- **Phase 3 — Backend collapse. DONE (2026-07-02).** New user drives default to **Autobase-from-birth** (the create-drive modal defaults to collaborative). The `serveAutobase` path reached parity with the Hyperdrive serve path — custom `.ui` frontend, directory index, **Range** on blobs (Phase 1), and now **markdown render + `.goto` redirects** (`.goto` href read from the file body, since Autobase has no entry metadata). The three real userland try/fallback dances (`reader` `openFeed`, `explorer`, `editor`) were collapsed onto `beaker.fs` (webterm's `hyper://private/` reads were repointed too). *Not done (see below):* physically making `createNewDrive`/the daemon stop minting Hyperdrives — the Hyperdrive backend is retained **internally** behind `beaker.fs` to still read `hyper://private/`, profile drives, and external Hyperdrives.
- **Phase 4 — Retire the split. LARGELY DONE (2026-07-02).** The public **`beaker.hyperdrive` and `beaker.autobase` APIs are removed** — RPC registrations dropped, `fg/hyperdrive.js`/`fg/autobase.js` + `manifests/external/{hyperdrive,autobase}.js` deleted, and **all ~63 userland call sites plus the fg-process RPC layer (modals/shell-menus/…) rewritten to `beaker.fs`**, which is now a full superset (files + drive lifecycle + writer management + import/export). The dts and AI reference no longer expose the two APIs. ADR-0007 aggregation code (`syncWriterKeys`, `ensureProfileContentDrive`, `writer-keys.json` + schema, `profileContentKey`) and ADR-0006 migration code (`migrateSpaceRootDrive`, `migrateAllSpaces`, the `beaker.vault.migrate` endpoint) are deleted. `CONTEXT.md`, `CLAUDE.md`, and the multi-device protocol doc collapse the Drive/Collaborative-Drive terminology. **Re-homing DONE (2026-07-02):** `hyper://private/` and every root/space drive are now created and loaded as **Autobase** — `filesystem/index.js` mints them via `autobases.createCollaborativeDrive`, loads via `getOrLoadCollaborativeDrive`, and reads/writes through backend-dispatch helpers (`_get`/`_put`/`_del`/`_entry`); `.goto` bookmarks moved to JSON bodies (the serve path reads the href from the body). `hyper.drives.getOrLoadDrive` + `getDriveCheckout` are Autobase-aware (return the loaded collaborative session / live view) so no caller opens an autobase core as a Hyperdrive and hangs. Consumers keyed on the sync Hyperdrive-only `getDrive()` (pane live-reload, permissions experimental-API gate) **degrade gracefully** (feature simply off for autobase drives). **Still deferred:** (a) `bg/hyperdrive.js` + `bg/autobase.js` remain as *internal* impls that `beaker.fs` still dispatches to for any non-autobase/external URL — physically deleting the Hyperdrive backend + `daemon.createHyperdriveSession` is the last cleanup, safe only once external-Hyperdrive support is formally dropped; (b) `shell` import/export and versioned checkout are degraded for autobase drives (a real Autobase checkout is Q5); (c) **Q4 encryption parity** for the now-Autobase private drive; (d) live-runtime verification of the whole FS-root re-home (unverifiable in this environment — a manual smoke run is required before shipping).
- **Phase 5 — Mobile parity. Code DONE; live gate open.** `mobile/backend/lib/drive-manager.mjs` + `vault.mjs` run on the shared `shared/fs-core.mjs` v1 format, and the app and mobile dependency trees produce a **byte-identical v1 view offline** (verified). The in-page drive API is also unified: the method list is the shared **`shared/fs-manifest.mjs`**, and mobile now exposes **`window.beaker.fs`** (replacing the old read-only `beaker.autobase`/`beaker.hyperdrive` shim) — `mobile/lib/types.ts` builds the surface, `mobile/backend/backend.mjs` `dispatchBeaker` handles `api:'fs'`. **Reads AND writes work** on mobile: `put`/`writeFile`/`del`/`mkdir`/`createDrive` route to `DriveManager`'s owned-drive path, with a persisted key→namespace registry (`drives-ns.json`) so the WebView bridge (which only sees a URL) can reopen an owned drive writable across restarts; a drive not owned on this device returns a clear read-only error. Still deferred on mobile: **per-drive writer management** (mobile's writer story is Vault device-pairing, not per-drive invites), `stat`/`entry`/`diff`, and — a genuine architectural blocker, **not** a wiring gap — **writes to drives paired-into but not created here.** `app/scripts/spike-shared-local-core.mjs` proves the device's single writer identity is the **exclusive** root `local` core the Vault holds, so opening a paired drive writable on the same corestore **deadlocks**; the fix is a **per-drive device writer key** (added via fan-out instead of the single `deviceKey`), which opens without deadlock but is a multi-device **protocol change** across both apps + pairing/fan-out, needing live-runtime verification (see multi-device-protocol.md §3). Mobile's `beaker.fs` therefore correctly **rejects** paired-into writes rather than hanging. Real desktop↔mobile **swarm** replication (incl. cross-writer blob resolution over the wire) and a device smoke-run of the shim remain the final acceptance gate — unbuildable/unrunnable in this environment.

Autobase drives currently serve the **live** view for `+version` (no point-in-time snapshot yet). A proper Autobase checkout rebuild (Hyperbee `view.checkout`, made robust to re-linearization) remains the flagged research spike.
